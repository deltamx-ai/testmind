# Jira 拆分任务场景下基于代码上下文的 LLM E2E 测试设计

创建时间：2026-03-21
项目目录：/home/delta/workspace/ai/testmind

## 一、问题本质

如果一个 Jira 对应的是完整功能需求，那么让 LLM 根据需求描述生成测试场景，难度还相对可控。

但真实研发流程里，很多 Jira 往往是拆分后的子任务，比如：

- 新增一个按钮
- 修改一个弹窗交互
- 调整某个字段校验
- 改一个接口返回后的状态展示
- 只改某个页面里的局部逻辑

这种情况下，如果只看 Jira 文本，LLM 最大的问题不是“不会写测试”，而是它根本不知道：

- 改动发生在哪个页面
- 用户从哪里进入这个功能
- 这个功能在完整链路里处于哪一步
- 相关页面元素是什么
- 要操作哪个按钮、表单、弹窗、列表项
- 什么是预期结果

所以答案是：要，必须结合代码，而且不能只结合一点点，最好要把“需求上下文 + 代码上下文 + 测试上下文”一起设计进去。

## 二、为什么必须结合代码

### 1. Jira 描述通常不够定位页面
很多 Jira 写法都是偏业务表达，比如“支持批量审批后显示审批结果”。
它说的是业务动作，但没有告诉你：

- 是哪个页面做批量审批
- 是列表页还是详情页
- 审批入口在顶部、行内还是弹窗里
- 成功提示是 toast、badge、tag 还是状态列变化

这些东西只有代码里更容易找到。

### 2. 页面操作路径往往藏在前端实现里
E2E 最关键的是“怎么进入这个功能”。
Jira 经常不会写完整路径，但代码里有：

- 路由配置
- 菜单配置
- 页面组件命名
- 按钮文案
- data-testid
- 调用的接口

这些信息能帮助 LLM 判断到底该从哪里开始跑测试。

### 3. 预期行为往往要结合代码和现有测试约定
Jira 说“支持审批成功后显示状态”，但状态显示成什么，不一定写得清楚。
代码和现有测试里更容易提取：

- 文案是什么
- 是否出现 toast
- 哪个 DOM 元素会变化
- 是否会跳转
- 是否会触发接口刷新

### 4. 拆分任务本身就是局部修改
拆分 Jira 往往不是全页面重做，而是局部改动。
这时最重要的是定位“受影响范围”，而不是泛泛生成整页测试。
这件事非常依赖代码变更信息。

## 三、推荐的设计思路

不要设计成：

Jira -> 直接生成 E2E

这个方案很容易失真。

更合理的是设计成一个多阶段链路：

Jira -> 需求结构化 -> 代码上下文定位 -> 候选页面/候选操作识别 -> 测试场景生成 -> E2E 脚本生成 -> 执行与反馈

也就是说，LLM 不应该一开始就负责“写最终测试脚本”，而应该逐层缩小范围。

## 四、建议的系统分层

### 第一层：Jira 解析层
这一层负责把 Jira 从自然语言描述转成结构化需求对象。

建议抽取的信息包括：

- Jira 标题
- Jira 描述
- 验收标准
- 关联需求/父任务
- 模块名
- 关键词
- 用户动作
- 预期结果
- 影响对象
- 可能的异常场景

输出示例可以是：

```json
{
  "feature": "批量审批",
  "action": "提交审批",
  "expected_result": "审批完成后状态更新",
  "module_keywords": ["审批", "列表", "状态"],
  "scope_hint": "列表页局部改动"
}
```

这一层的目标不是生成测试，而是先把需求讲清楚。

### 第二层：代码定位层
这一层是整个方案成败的关键。

它的目标是根据 Jira 结构化结果，到代码里找到最可能相关的地方。

可以利用的信息包括：

- 路由文件
- 页面组件名
- 模块目录名
- 文案常量
- i18n key
- 接口名
- 提交记录
- PR diff
- 最近变更文件
- 现有测试文件
- 埋点名
- data-testid

这一层最终要输出的不是“唯一答案”，而是“候选答案集合”，例如：

- 候选页面 A：/approval/list
- 候选组件 B：BatchApproveDialog
- 候选操作 C：点击顶部“批量审批”按钮
- 候选断言 D：状态列从 pending 变为 approved

注意，必须允许不确定性，不要假装自己 100% 知道。

### 第三层：测试意图生成层
在知道候选页面和候选操作后，再由 LLM 生成测试意图。

这里生成的不是最终代码，而是：

- 覆盖哪些用户路径
- 哪些前置条件必需
- 哪些断言最关键
- 哪些异常场景值得补
- 哪些地方适合做回归保护

例如：

- 正常流程：选择多条记录 -> 点击批量审批 -> 确认 -> 状态更新
- 异常流程：未选择记录时点击审批 -> 给出提示
- 权限场景：无审批权限用户不显示按钮
- 数据场景：部分不可审批记录时的处理方式

这一层像“测试设计师”。

### 第四层：E2E 生成层
到这一层，才开始生成具体 E2E 脚本。

这里建议一定要建立模板化生成，而不是让 LLM 完全自由发挥。

比如项目统一使用 Playwright，那么可以给它固定约束：

- 使用统一目录结构
- 使用统一 page object
- 使用统一登录方法
- 使用统一数据准备方法
- 优先使用 data-testid
- 使用统一断言风格
- 输出注释说明脚本来源于哪个 Jira

也就是说，LLM 更像是往既有模板里填内容，而不是从零写一整套测试框架。

### 第五层：执行反馈层
脚本生成后，跑起来才有意义。

这里要输出：

- 通过/失败状态
- 失败步骤
- 页面截图
- DOM 快照
- 网络请求日志
- 控制台错误
- 失败归因建议

如果没有这一层，整个系统就只是“自动写测试代码”，还谈不上真正帮助开发提前发现 Bug。

## 五、一个实际可落地的设计方案

### 方案核心
设计一个“Jira + 代码索引 + LLM 推理”的混合系统，而不是纯 LLM 系统。

具体来说：

#### 1. 先建立代码索引
提前扫描项目，构建可供检索的上下文索引，例如：

- 页面与路由映射
- 页面与接口映射
- 页面与组件映射
- 页面与 data-testid 映射
- 页面与菜单入口映射
- 现有测试与页面映射

这一步相当于给 LLM 建一张“地图”。

#### 2. Jira 输入后先做检索，不直接生成
拿 Jira 里的关键词去索引里找候选页面和候选操作。

比如“审批”“批量”“状态”，先检索出相关页面、组件、测试文件、接口，再把这些上下文喂给 LLM。

这样 LLM 就不是凭空猜，而是“带参考资料作答”。

#### 3. 生成中间产物
不要一步到最终脚本，建议先输出三个中间结果：

- 需求理解结果
- 代码定位结果
- 测试场景结果

人可以 review 这三步中的任何一步。

#### 4. 最后再生成脚本
只有当前三步置信度足够高时，再输出 Playwright/Cypress 脚本。

#### 5. 执行结果反哺索引
如果脚本执行失败，可以把失败结果回写，帮助后续识别：

- 哪些 selector 不稳定
- 哪些页面路径判断不准
- 哪些类型 Jira 难以自动生成

这样系统会越来越准。

## 六、输入应该有哪些

如果你真要做这个方案，建议输入不要只有 Jira 文本，而应该尽量包含：

- Jira 标题
- Jira 描述
- 验收标准
- 父任务/史诗
- 关联设计稿
- 关联 PR
- 变更文件列表
- 变更 diff
- 路由表
- 页面组件索引
- 现有自动化测试
- data-testid 清单
- 测试账号和数据准备规则

这里面价值最高的其实不是 Jira 描述，而是：

- 代码 diff
- 路由映射
- 现有测试资产
- 可定位元素信息

## 七、输出应该长什么样

建议系统输出不要只有测试脚本，而是分层输出。

### 输出一：需求理解
说明 LLM 认为这个 Jira 在改什么。

### 输出二：代码定位
说明它认为相关页面、组件、接口是什么，并给出置信度。

### 输出三：测试场景
说明应该覆盖哪些路径，哪些是主路径，哪些是异常路径。

### 输出四：E2E 脚本
只有在定位足够明确时才生成。

### 输出五：风险提示
例如：

- 页面定位不明确
- 缺少稳定 selector
- 缺少测试数据准备方式
- Jira 描述信息不足

这一步很重要，因为它能避免系统“假装很懂”。

## 八、怎么让它知道是哪个页面、执行什么操作

这个问题可以直接回答：

不是“让 LLM 自己知道”，而是“通过检索和约束让它推断出来”。

具体依赖四类信息：

### 1. 关键词匹配
Jira 里的业务词和代码里的页面/组件/接口名称做匹配。

### 2. 变更感知
读取 PR diff 或变更文件，优先把发生改动的页面作为候选。
这个非常关键，比纯文本分析靠谱得多。

### 3. 页面地图
提前维护系统的页面、路由、入口、关键操作映射。

### 4. 现有测试资产复用
如果某个页面已经有旧 E2E 或 page object，新的测试最好在原基础上扩展，而不是重新瞎猜。

## 九、最小可行版本怎么做

如果现在就要落地，建议先做最小版本，不要做太重。

### MVP 建议
先支持这样一条链路：

Jira + PR diff -> 推断受影响页面 -> 生成人工可 review 的测试点 -> 生成 Playwright 草稿 -> 开发手动确认后执行

这个版本先解决三个问题：

- 帮开发快速补测试点
- 帮开发快速写 E2E 草稿
- 降低纯手工写测试的成本

先不要追求：

- 全自动确定页面
- 全自动稳定执行
- 全自动识别所有 Bug 原因

这些都是后期增强项。

## 十、结论

所以这个问题的答案很明确：要结合代码，而且最好不是简单“读一下代码”，而是把代码上下文设计成系统输入的一部分。

真正靠谱的方案不是“Jira -> LLM -> E2E”，而是：

Jira -> 结构化需求 -> 检索代码上下文 -> 定位候选页面和操作 -> 生成测试场景 -> 生成 E2E -> 执行反馈

这样设计的好处是：

- 不靠大模型瞎猜
- 能处理拆分 Jira
- 能把不确定性显式表达出来
- 能和现有前端项目结构、测试框架结合
- 更适合逐步落地
