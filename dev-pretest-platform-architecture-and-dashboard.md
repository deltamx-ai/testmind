# 面向开发自测的提测质量平台技术选型与总览面板方案

创建时间：2026-03-21
项目目录：/home/delta/workspace/ai/testmind

## 一、这次要补的核心问题

如果前一版方案解决的是“为什么做、做什么、怎么减少提测 Bug”，那么这一版要解决的是更落地的两个问题：

第一，这个东西应该做成什么产品形态。

第二，既然给开发使用，但又希望整体能看统计数据，那系统的展示层应该怎么设计。

所以这次要回答的是：

到底做前后端分离的平台，还是做一个 Electron 桌面端；
开发怎么用；
管理和团队怎么查看总体效果；
系统里应该沉淀哪些统计数据。

## 二、先说结论

如果目标用户是开发人员，但同时希望沉淀跨项目数据、跨团队统计和统一展示面板，那么更推荐做“前后端分离的平台”，而不是把它做成一个纯 Electron App。

更准确地说，最合适的形态是：

一个以 Web 平台为核心的质量分析与自测辅助系统，必要时再补一个轻量本地接入层或者 IDE / CLI 入口，而不是从一开始就做成重桌面应用。

原因很简单。

Electron 更适合强调本地体验、单人使用、强本地集成的工具。
但你这里的目标已经不只是“开发本地跑一下”，而是要：

- 识别多个项目
- 汇总多项目测试数据
- 展示测试通过率和失败分布
- 让不同开发、不同团队都能使用
- 给团队或负责人提供统一总览面板

这些需求天然更像平台型产品，而不是单机工具。

## 三、为什么不建议一开始做成纯 Electron

Electron 不是不能做，而是它不适合作为第一优先形态。

因为如果做成纯桌面 App，会遇到几个问题。

### 1. 数据天然分散
每个开发本地跑出来的数据都散落在自己的机器上。

这样虽然能帮单个开发自测，但很难统一看到：

- 当前有哪些项目在接入
- 每个项目有多少测试用例
- 哪些模块失败率高
- 哪些 case 通过率最低
- 哪个团队的提测质量在改善

而这些恰恰是你现在明确希望收集和统计的东西。

### 2. 总览面板不好做
如果老板或者负责人想看整体情况，纯 Electron 架构会导致统计能力比较别扭。

因为你要么做本地汇总上传，要么做复杂同步机制。
这相当于最后还是会逼你做一个中心化服务。

既然中心化服务迟早都要做，那不如一开始就按平台架构设计。

### 3. 运维和升级成本更高
桌面应用意味着：

- 安装和升级要管理
- 多平台兼容要处理
- 本地环境差异多
- 本地权限问题更复杂
- 调试成本更高

如果系统还要和 Jira、Git、CI、测试框架打通，Web 平台加本地 agent 的方式通常会更稳。

### 4. 对团队协作不友好
这个项目不是纯个人效率工具，它更像“个人使用 + 团队管理 + 组织数据沉淀”的混合型产品。

这种产品核心一定要有服务端，前端只是入口。

## 四、推荐的产品形态

我更推荐下面这个组合。

### 1. 核心形态：前后端分离 Web 平台
系统核心做成一个标准的前后端分离平台。

前端负责：

- 开发查看自己项目的自测建议
- 查看测试结果和失败摘要
- 查看项目统计与趋势
- 查看团队/项目总览面板

后端负责：

- 接 Jira、Git、CI、测试结果
- 做项目、用例、执行记录和失败原因的结构化存储
- 调 LLM 做分析、归纳、生成
- 产出统计指标和 dashboard 数据

这部分是整套系统的主干。

### 2. 本地接入层：CLI 或轻量 Agent
虽然平台是 Web 为核心，但开发的实际使用动作很可能发生在本地。

所以建议配一个轻量入口，比如 CLI。

开发可以在本地执行类似：

提交当前 Jira 和分支信息 -> 触发平台分析 -> 拉取自测建议 -> 可选地在本地执行测试 -> 上传结果

这样开发体验不会差，同时平台也能拿到中心化数据。

### 3. 后续增强：IDE 插件或桌面壳
如果后面发现开发特别在意本地体验，可以再补一个 IDE 插件，或者套一个 Electron 壳。

但那应该是交互层增强，不应该是底层架构起点。

也就是说，Electron 可以有，但更适合作为后期补充入口，而不是第一阶段主形态。

## 五、建议的技术选型

如果按平台思路来做，技术上建议偏稳一点，不要花太多力气在形态争议上。

### 前端建议
前端建议直接做 Web 控制台。

如果团队已有 React 技术栈，就继续用 React。
为了提高开发效率，可以配：

- React + TypeScript
- Next.js 或 Vite + React Router
- 组件库用 Ant Design 或者已有内部设计体系
- 图表层用 ECharts 或 AntV

如果这个系统需要比较强的表格、筛选、趋势图、钻取能力，React 生态会比较顺手。

### 后端建议
后端建议做成标准 API 服务。

如果团队偏 Node.js，可以用：

- NestJS 或 Express / Fastify
- TypeScript
- PostgreSQL 存结构化业务数据
- Redis 做任务状态、缓存和队列辅助

如果团队偏 Java 或 Go，也可以，但从“要接 LLM、要做 JSON 流式结果、要快速迭代”的角度，Node.js / TypeScript 会更灵活。

### 异步任务建议
这个系统一定会有异步任务，比如：

- 拉 Jira 信息
- 拉 PR diff
- 检索代码上下文
- 调 LLM 分析
- 生成测试草稿
- 接收测试执行结果
- 统计聚合更新 dashboard

所以建议用任务队列，不要全塞进同步 API。

Node 侧可以考虑 BullMQ 这类方案。

### 数据存储建议
这个系统最少要存四类核心数据：

项目信息、测试资产信息、执行记录、质量统计结果。

比较合理的结构包括：

- Project：项目基本信息
- JiraTask：需求任务信息
- ChangeSet：PR / diff / 变更文件
- TestCase：测试用例或测试草稿
- TestExecution：每次执行记录
- FailureIssue：失败问题摘要
- QualityMetric：按项目、时间、模块聚合的统计指标

这类数据天然适合放 PostgreSQL。

如果后面要做语义检索，再补向量索引或检索服务就行，不用一开始搞得太重。

## 六、系统架构建议

推荐采用“平台中心 + 本地执行”的混合架构。

整体链路可以这样理解：

开发在 Web 或本地 CLI 发起一次提测前检查。
平台拉取 Jira、PR diff 和项目配置。
后端分析本次改动影响范围。
系统生成自测建议和测试草稿。
如果项目支持自动执行，就触发本地或 CI 执行。
执行结果再回传平台。
平台统一生成问题摘要、项目统计和全局 dashboard。

这个架构有几个好处：

既照顾开发本地使用习惯，也保留中心化统计和管理能力。

## 七、既然要收集统计信息，系统里应该有哪些核心对象

你提到的几个统计问题其实已经很关键了，比如：

有哪些项目；
多少测试用例；
哪个 case 通过率最低最高。

基于这些需求，系统里至少要有下面这些对象。

### 1. 项目维度
每个接入系统的研发项目都要有自己的档案。

要记录：

- 项目名称
- 仓库地址
- 技术栈
- 测试框架
- Jira 项目标识
- 接入状态
- 最近执行时间
- 项目负责人

这样平台才能回答“现在接入了哪些项目”。

### 2. 测试用例维度
无论是已有用例还是新生成草稿，都应该统一管理。

要记录：

- 用例名称
- 所属项目
- 所属模块
- 来源是已有用例还是 AI 生成
- 对应 Jira / 需求
- 最近执行结果
- 历史通过率
- 最近失败原因分布

这样平台才能回答“有多少测试用例”“哪些 case 最稳定、哪些最不稳定”。

### 3. 执行记录维度
每次执行都应该被保存。

要记录：

- 执行时间
- 执行人或触发方式
- 对应分支/commit
- 执行了哪些 case
- 通过数、失败数、跳过数
- 失败截图、日志、归因摘要

这部分是所有统计分析的基础。

### 4. 质量指标维度
为了做 dashboard，平台还要做聚合指标。

例如：

- 项目测试通过率
- 模块失败率
- case 稳定性排名
- 提测前拦截问题数
- 提测后问题回流数
- 生成建议采纳率

## 八、总览面板应该怎么设计

既然主要用户是开发，但又希望整体有一个展示面板，那最合适的是做两层面板。

### 第一层：开发视角面板
这是给单个开发或单个项目成员看的。

重点不是全局管理，而是“我现在这次提测有没有风险”。

建议展示：

- 本次 Jira / 分支对应的影响范围
- 建议自测项
- 推荐复用的已有用例
- 新生成测试草稿
- 最近一次执行结果
- 当前失败问题摘要

这个页面偏操作型，帮助开发完成当下这次提测。

### 第二层：全局总览面板
这是你说的总体展示面板。

这个页面更偏团队管理和平台价值展示。

建议首页重点展示这些。

先看项目总体情况：

当前已接入多少项目，哪些项目接入完成，哪些项目最近活跃。

再看测试资产规模：

总用例数、近 7 天新增用例数、AI 生成草稿数、活跃执行用例数。

再看质量表现：

整体通过率、各项目通过率排名、失败率最高项目、最近问题波动趋势。

再看 case 稳定性：

通过率最高的 case、通过率最低的 case、失败波动最大的 case、长期未执行的 case。

最后看业务价值：

提测前拦截的问题数、提测后被测试发现的问题数变化、自测建议采纳率。

这样一来，开发有自己的工作面板，负责人也有自己的总览面板。

## 九、你点名要看的几个统计项，具体怎么展示

你提到的几个指标非常适合直接上首页。

### 1. 有哪些项目
这块建议做成项目列表 + 状态卡片。

每个项目卡片显示：

- 项目名
- 测试框架
- 接入状态
- 总用例数
- 最近 7 天执行次数
- 最近通过率

这样很直观。

### 2. 多少测试用例
这块建议显示两个数字：

一个是总用例数；
一个是近 7 天活跃执行的用例数。

因为只有总数没有意义，活跃度才说明系统是不是在真实使用。

### 3. 哪个 case 通过率最低、最高
这块建议做两个榜单：

一个是“最稳定 case Top N”；
一个是“最不稳定 case Top N”。

并且每个 case 最好能点进去看：

- 所属项目
- 所属模块
- 历史执行次数
- 最近失败原因
- 关联 Jira / 变更

这样这个榜单不只是展示，也能指导治理。

## 十、这套系统最终更像什么

如果把产品形态说得更直白一点，它更像：

一个面向研发提测前场景的质量平台。

它不是单纯本地工具，因为它要做数据沉淀和跨项目统计；
它也不是纯管理后台，因为开发本人还要真正在里面拿建议、跑检查、看失败摘要。

所以最合适的理解就是：

平台化系统 + 本地执行入口 + 全局质量面板。

## 十一、实施建议

如果现在马上要推进，我建议先这样分阶段。

第一阶段先做 Web 平台骨架和项目接入能力，把项目、用例、执行记录、结果回传这些主数据链路打通。

第二阶段做开发可用的提测前检查入口，可以先是 Web 页面触发，也可以是 CLI 触发。

第三阶段做 LLM 分析、自测建议生成和失败摘要。

第四阶段再强化 dashboard 和全局统计能力。

如果后面确实证明开发很需要桌面体验，再考虑补 Electron 壳，但这时它只是一个入口，不是主系统。

## 十二、结论

结合你的目标，这个系统不建议优先做成纯 Electron App。

更推荐做成前后端分离的 Web 平台，并补一个本地 CLI 或轻量 agent 作为开发入口。

这样既能满足开发日常使用，也能满足你想要的跨项目统计、用例规模统计、case 通过率排名和统一展示面板。

一句话收敛就是：

产品形态选平台，不选单机；
交互入口可多样，但数据和分析必须中心化。
