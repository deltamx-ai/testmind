# 面向隐私优先研发自测工具的可扩展架构设计

创建时间：2026-03-21
项目目录：/home/delta/workspace/ai/testmind

## 一、架构目标

既然当前方向已经更偏向 Electron 本地优先，那么架构设计的重点就不应该只是“把功能做出来”，而是要保证后面继续扩功能时不会越做越乱。

这个系统后续大概率还会不断加东西，比如：

- 新的模型接入方式
- 新的测试框架支持
- 新的统计面板
- 新的项目接入方式
- 新的认证方式
- 新的执行器
- 新的插件能力
- 新的团队协作能力

所以现在设计架构时，最重要的目标不是极致简单，而是“核心稳定、边界清楚、扩展低成本”。

换句话说，这套架构要满足四件事：

第一，本地优先，敏感数据默认留在本机。

第二，模块解耦，新增功能尽量不改核心流程。

第三，能力可插拔，模型、执行器、数据源、展示层都能扩展。

第四，未来如果要增加少量中心化能力，也不要推翻重来。

## 二、先说整体思路

我更推荐把这个 Electron 应用设计成“壳 + 核心引擎 + 插件能力”的结构。

不要把所有业务逻辑都塞进 Electron 页面层，也不要把测试执行、模型调用、项目解析这些能力直接耦合在一起。

更合理的做法是：

Electron 负责桌面壳、界面交互和本地能力桥接；
核心引擎负责任务编排、领域流程和状态管理；
插件层负责接入不同的数据源、模型、测试执行器和扩展功能。

这样一来，Electron 只是产品载体，不会变成业务黑洞。

## 三、推荐的总体分层

整体建议拆成六层。

### 第一层：桌面壳层
这层就是 Electron 本身。

负责：

- 窗口管理
- 本地菜单
- 系统托盘
- 文件系统访问桥接
- 本地权限控制
- 渲染进程和主进程通信
- 本地通知

这一层不要放太多业务逻辑。
它的职责是承载应用，不是承载业务规则。

### 第二层：应用界面层
这层就是 React 或其他前端界面。

负责：

- 项目选择
- 提测检查入口
- 自测建议展示
- 执行结果展示
- case 统计面板
- 设置页
- 插件管理页

这一层只处理交互和展示，不直接耦合底层实现细节。

也就是说，界面不应该直接知道 Playwright 怎么跑，也不应该直接知道 Jira 怎么拉。

### 第三层：应用服务层
这是整个系统里最关键的一层之一。

它负责把界面动作变成稳定的业务用例。

比如：

- 发起一次提测前检查
- 生成某次变更的自测建议
- 执行一组选中的测试
- 读取某个项目的历史统计
- 同步某个 Jira 任务信息
- 更新本地模型配置

这一层相当于应用 orchestration 层。

它只面向业务动作，不面向底层实现。

### 第四层：领域核心层
这一层是最值得保护的核心。

它应该沉淀系统真正稳定的领域模型和流程规则，比如：

- 项目 Project
- 变更 ChangeSet
- 自测任务 CheckTask
- 测试用例 TestCase
- 执行记录 Execution
- 问题摘要 IssueSummary
- 建议结果 Recommendation
- 指标快照 MetricSnapshot

以及这些对象之间的核心关系和状态流转。

为什么这层重要。

因为以后你无论接 Jira、GitLab、Azure DevOps，还是换模型、换执行器，只要领域核心不乱，系统就不会散架。

## 四、把变化点放在边缘，不要放在中心

要想易扩展，最重要的原则就是：

稳定的放中间，变化的放边缘。

在这个系统里，最稳定的是：

- 提测前检查这条业务主线
- 项目、变更、测试、执行、结果这些核心对象
- 本地优先和最小上传这条隐私原则

最容易变化的是：

- Jira 来源可能会变
- Git 平台可能会变
- LLM 模型可能会变
- 测试框架可能会变
- 统计需求可能会变
- 认证方式可能会变

所以正确架构不是把这些变化点写死，而是全部抽成接口或插件点。

## 五、最应该插件化的四类能力

我建议最少把下面四类能力做成可替换组件。

### 1. 数据源插件
数据源不应该只支持 Jira。

后面可能还会接：

- Azure DevOps Work Item
- GitHub Issues
- GitLab Issue
- 本地 markdown 需求文件
- PR 描述

所以数据源这一层应该抽象成统一接口。

输入是某个任务标识或上下文，输出是统一结构化需求对象。

这样以后加新来源，不需要改主流程。

### 2. 代码上下文提供器
代码上下文获取方式也会变化。

有的项目可能依赖：

- git diff
- 路由文件扫描
- 页面目录约定
- 测试目录约定
- AST 分析
- 组件索引

所以这层也要抽象。

主流程只关心“给我当前任务对应的代码上下文”，不关心它到底是扫文件还是读索引。

### 3. 模型提供器
模型层一定要抽象，不要把某个模型 SDK 写死。

后面很可能会同时存在：

- 本地模型
- 企业内网模型
- 外部 API 模型
- Copilot 协作入口

所以模型接口建议统一成类似：

输入上下文、任务类型、输出模式；
返回结构化结果、置信度、可选解释。

这样无论换模型还是多模型路由，都不会影响上层业务逻辑。

### 4. 执行器插件
测试执行也不要只写死 Playwright。

虽然当前最推荐 Playwright，但后续可能会出现：

- Cypress
- 接口测试执行器
- 移动端执行器
- 仅静态检查执行器

所以执行器层应该统一抽象成：

接受测试任务，返回执行结果、附件、错误信息、指标。

这样以后你扩测试类型时不会重写整套系统。

## 六、推荐的目录和模块组织方式

如果落地到代码结构，我建议按 monorepo 思路来组织，即使一开始功能不多，也值得这样做。

可以按下面这种思路。

apps/desktop
放 Electron 主进程、预加载脚本、桌面生命周期管理。

apps/ui
放 React 界面层，包括项目页、任务页、结果页、统计页、设置页。

packages/core
放领域模型、核心流程、业务规则、任务状态机。

packages/application
放用例编排和应用服务。

packages/plugin-sdk
放插件接口定义，比如数据源、模型、执行器、认证、存储扩展点。

packages/plugins
放具体插件实现，比如 jira-plugin、git-plugin、playwright-plugin、local-llm-plugin。

packages/storage
放本地数据库 schema、repository、迁移脚本。

packages/shared
放公共类型、日志、工具函数、事件定义。

这种结构的好处是：

后面新增能力时，大多数情况只是在 plugins 或 application 层加东西，不会把 desktop 和 ui 改成一团。

## 七、数据存储也要为扩展留口子

既然是本地优先，数据存储建议默认走本地数据库。

最合适的是 SQLite。

原因很简单：

- 本地部署轻
- 不需要额外服务
- 适合 Electron
- 查询能力足够
- 容易做迁移和备份

建议把数据也分层。

### 本地主数据
用于保存：

- 项目配置
- 需求任务缓存
- 代码分析结果
- 测试用例索引
- 执行记录
- 问题摘要
- 指标快照
- 用户设置

### 本地附件数据
用于保存：

- 截图
- trace
- 视频
- 原始报告
- 调试日志

不要把大文件塞数据库里，数据库里只存路径和索引。

### 可选同步数据
如果以后加中心服务，就只同步少量脱敏聚合数据。

这一层在架构上提前留接口，但默认可以不启用。

## 八、系统内部通信建议事件化

如果想让系统后面容易加功能，一个很有效的办法是让内部通信尽量事件化，而不是大量模块彼此直接调用。

比如这些事件就很有用：

- project.added
- changeset.detected
- recommendation.generated
- testcase.generated
- execution.started
- execution.finished
- issue.summary.created
- metric.snapshot.updated

这样做的好处是，后面新增功能时，很容易通过订阅事件挂进去。

比如你未来想加：

- 执行完成后自动更新面板
- 失败后自动生成修复建议
- 每周自动汇总本地趋势
- 可选上传脱敏统计

都不需要改主流程，只要加一个事件处理器就行。

## 九、怎样保证以后加功能不把系统搞乱

这里有几个很关键的架构纪律。

第一，不要让 UI 直接调底层插件。
UI 只能调 application service。

第二，不要让插件知道太多业务细节。
插件只负责能力提供，不负责业务编排。

第三，不要把模型输出直接当数据库结构。
模型结果先转成领域对象，再落库。

第四，不要把统计逻辑写死在执行流程里。
统计应该通过事件或异步聚合生成。

第五，不要把隐私策略散落在各处。
应该有统一的数据分级和上传策略控制点。

这几条守住了，后面加功能会顺很多。

## 十、推荐加入一个能力注册中心

如果要方便以后加功能，我建议系统里有一个注册中心或者 capability registry。

它的作用是统一声明当前应用有哪些可用能力，比如：

- 当前有哪些数据源插件
- 当前有哪些模型提供器
- 当前有哪些执行器
- 当前有哪些同步策略
- 当前有哪些统计扩展

这样在设置页、插件管理页、调试页里都能统一展示，也方便后面做按项目配置。

例如某个项目可以选择：

- 需求源用 Jira
- 模型用企业内网模型
- 执行器用 Playwright
- 同步策略设为关闭

另一个项目则可以选完全不同的组合。

这会让系统扩展性非常强。

## 十一、如果未来要从单机走向轻协作，怎么演进

这套架构还要考虑未来演进，不然一开始选本地优先，后面又容易重构。

比较好的演进方式是：

第一阶段，单机本地闭环。
Electron + SQLite + 本地插件。

第二阶段，可选聚合。
增加一个同步插件，只上传脱敏指标。

第三阶段，轻协作。
增加团队共享规则、共享模板、共享插件配置。

第四阶段，混合架构。
本地执行保持不变，中心服务只做低敏聚合和组织视图。

注意这条演进路线里，本地能力始终是主角，中心能力只是外挂。

这样就不会背离“隐私优先”的初衷。

## 十二、最推荐的架构结论

如果让我给一句收敛后的架构建议，我会这样定。

把这个系统设计成：

一个基于 Electron 的本地优先桌面应用，采用“桌面壳 + 应用服务 + 领域核心 + 插件层 + 本地存储 + 事件总线”的可扩展架构。

其中：

- Electron 负责桌面壳和本地能力桥接
- React 负责交互和面板展示
- Application 层负责业务用例编排
- Core 层负责稳定领域模型和主流程
- Plugin 层负责数据源、模型、执行器、同步能力扩展
- SQLite 负责本地主数据持久化
- 附件文件系统负责截图、trace、日志保存
- Event Bus 负责模块解耦和后续能力挂载

这套架构既适合当前隐私优先方向，也能保证后续想加能力时不会牵一发动全身。

## 十三、结论

真正易扩展的关键，不是把系统拆得越多越好，而是把“稳定的核心”和“高变化的能力”分开。

对你这个场景来说，最稳的方式不是做一个巨大的 Electron 页面应用，而是做一个有清晰中台核心的本地产品。

这样以后无论你加 Copilot、加新模型、加新测试框架、加本地趋势面板、加可选同步、加团队模板，都能在现有架构上平滑往前走。
