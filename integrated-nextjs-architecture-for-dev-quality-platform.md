# 面向研发提测质量平台的一体化技术方案

创建时间：2026-03-21
项目目录：/home/delta/workspace/ai/testmind

## 一、这次要回答的几个关键问题

现在问题已经不是做不做平台，而是平台到底怎么落地更合适。

你问的几个点其实很关键：

是不是直接用 Next.js；
前后端是不是可以不拆开，做成大一统；
测试执行是不是用无头浏览器；
如果要长期统计数据，怎么做持久化；
系统要不要登录；
如果公司已经有统一微软账号认证，怎么接进去。

这些问题合起来，其实就是这套系统的正式技术架构。

## 二、先说结论

如果当前目标是尽快落地一个可用的平台，而且前后端都由同一团队维护，我更推荐采用“大一统”的一体化 Web 架构。

具体建议是：

以 Next.js 作为主框架，前端界面、服务端接口、页面路由、部分服务端逻辑放在一个工程里；
测试执行能力、异步任务调度和浏览器执行器作为独立 worker 或 job service 存在；
数据统一持久化到中心数据库；
登录接公司现有微软统一认证。

一句话概括就是：

主系统用 Next.js 一体化开发，执行器独立，数据中心化，认证走企业微软账号。

这会比“完全前后端分离 + 再做很多基础设施”更轻，也比“纯本地工具”更适合你现在的目标。

## 三、为什么推荐 Next.js，而不是完全拆开的前后端

如果从工程效率和落地速度来看，Next.js 很适合这个项目。

因为这个系统本质上有三类需求同时存在：

第一类是标准 Web 界面，比如项目列表、任务详情、测试结果页、dashboard。

第二类是服务端逻辑，比如接收 Jira 信息、写数据库、聚合指标、触发任务。

第三类是少量需要实时返回或者流式展示的能力，比如 LLM 分析进度、任务执行状态、失败摘要回传。

Next.js 对这三类事情的兼容性都还不错。

尤其如果你们团队前端本来就熟 React，这个方案能明显降低沟通和开发成本。

而且你说“前后端都有，可以不用分离开，大一统方式”，这其实正好适合 Next.js 这种一体化风格。

## 四、推荐的整体形态

建议整体拆成三个层次，但代码组织上仍然可以是“大一统主工程 + 外围执行组件”。

### 第一层：Next.js 主应用
这部分是系统核心。

负责：

- 所有 Web 页面
- 登录态管理
- 项目管理
- Jira / 任务详情展示
- 自测建议展示
- 测试结果展示
- Dashboard 数据展示
- API 路由
- 部分轻量服务端逻辑

也就是说，用户能看到的大部分东西都在这里。

### 第二层：异步任务层
因为很多事情都不适合同步在请求里做，比如：

- 拉 Jira
- 分析 PR diff
- 调 LLM
- 生成测试草稿
- 跑 Playwright
- 汇总执行结果
- 刷新聚合统计

所以建议做任务队列和 worker。

这部分可以和 Next.js 放同仓库，也可以作为独立服务运行。

但逻辑上最好分开，不要把长任务直接塞进页面请求。

### 第三层：测试执行层
测试执行最好单独作为 runner 存在。

因为浏览器执行环境、依赖安装、账号配置、截图和视频落盘这些东西，和普通 Web 请求完全不是一类负载。

这层可以部署在 CI 节点、专用 runner 机器，或者容器化执行环境里。

## 五、测试执行是不是用无头浏览器

结论是：要，而且默认就是无头浏览器执行。

如果目标是帮助开发在提测前自动化验证页面行为，那最现实的方式就是浏览器自动化。

在 Web 场景里，优先推荐 Playwright。

原因很简单。

它对现代前端项目支持比较完整，支持 Chromium、Firefox、WebKit，支持无头模式，也支持截图、录屏、trace、网络日志、控制台日志，这些都非常适合你这个“提测前发现问题”的场景。

所以推荐做法是：

默认使用 Playwright 的 headless 模式执行测试；
如果需要调试，再允许切换成 headed 模式做人工复现。

这样既保证自动化效率，也方便排查问题。

## 六、为什么无头浏览器适合这个方案

因为你不只是想跑测试，而是想留下证据并做统计。

无头浏览器很适合做这些事：

- 自动登录
- 自动点击、输入、断言
- 保存截图
- 保存视频
- 保存 trace
- 记录控制台错误
- 记录网络请求失败
- 统一上报执行结果

这些结果再回到平台后，就能变成：

- 本次提测前通过了哪些检查
- 哪一步失败了
- 失败时页面长什么样
- 历史上这个 case 稳不稳定

所以测试执行层最好从一开始就按浏览器自动化来设计，而不是用一些零散脚本拼。

## 七、数据持久化怎么做

既然你希望收集统计信息，那数据一定不能只存在内存里，也不能只放在测试报告文件中。

必须做中心化持久化。

推荐思路是：

业务主数据进 PostgreSQL；
缓存、任务状态、短期队列数据进 Redis；
截图、视频、trace、报告文件进对象存储或者文件存储。

### 1. PostgreSQL 存什么
最关键的结构化数据建议都放 PostgreSQL。

至少包括：

- 用户信息
- 项目信息
- Jira 任务记录
- 代码变更记录
- 测试用例信息
- 测试执行记录
- 失败问题摘要
- dashboard 聚合指标
- AI 生成结果元数据

因为这些东西都需要筛选、排序、聚合、关联查询。

### 2. Redis 存什么
Redis 主要用来做：

- 任务队列
- 执行状态缓存
- 短期 session/cache
- 临时结果加速

不要把核心业务数据只放 Redis。
Redis 适合加速，不适合做长期真源。

### 3. 文件类结果放哪里
测试执行会产出很多附件，比如：

- 截图
- 视频
- trace 文件
- 原始 HTML 快照
- 测试报告文件

这些不适合直接塞数据库。

建议放对象存储，或者至少放统一文件目录，然后数据库里只记录路径和索引信息。

## 八、需要登录吗

需要，而且建议从第一版就接入登录。

原因不是为了“看起来正式”，而是这个系统本身就涉及权限边界。

比如：

- 不同开发能看到哪些项目
- 谁能查看失败详情
- 谁能看全局 dashboard
- 谁能配置 Jira / Git / 测试账号
- 谁能重新执行测试

如果没有登录和权限体系，后面一定会返工。

尤其你希望有总览面板，那就说明不只是一个个人本地工具，而是团队级平台。

团队级平台就必须有用户身份和权限。

## 九、公司已经有微软统一账号认证，怎么接

这个场景其实很适合，不要自己造登录系统。

推荐直接接公司现有的 Microsoft Entra ID，也就是微软统一身份认证。

Next.js 这类应用接企业 SSO 很成熟。

比较推荐的思路是：

使用标准 OAuth 2.0 / OpenID Connect 方式接微软登录；
登录成功后拿到用户身份信息；
系统再基于邮箱、部门、用户组或项目成员关系做本地授权。

也就是说，认证交给微软，授权由你自己的系统控制。

这样做的好处是：

用户体验统一，不需要单独记密码；
员工入离职可以复用公司身份体系；
权限治理更容易接入组织架构；
安全性也比自建账号密码高得多。

## 十、登录后的权限建议怎么设计

即使认证走微软，系统内部仍然要有自己的角色体系。

建议至少分成这几类角色：

普通开发，可以看自己有权限项目的自测结果、建议和 case 执行情况。

项目负责人，可以看项目级 dashboard、项目成员数据、项目配置。

测试或质量负责人，可以看跨项目质量统计、失败分布、提测问题趋势。

平台管理员，可以管理接入配置、模型策略、执行器、权限映射。

这样未来扩展会顺很多。

## 十一、Next.js 一体化架构下，数据和任务怎么流转

可以把流程理解成下面这样。

用户登录系统后，在某个项目里发起一次提测前检查。
Next.js 服务端收到请求，先写入一条任务记录到数据库。
然后把任务投递到队列。

Worker 拿到任务后：

去拉 Jira 信息；
去拉代码 diff；
做影响分析；
调用 LLM 生成自测建议；
如果需要就生成 Playwright 草稿；
再触发测试 runner 执行；
执行完成后把结果、截图路径、失败摘要都写回数据库。

最后，Next.js 页面从数据库读取状态并实时展示。

这样做有一个很大好处：

前台体验简单，但后台链路足够稳，而且所有数据天然可追踪、可统计。

## 十二、这种一体化方案适不适合长期扩展

适合，只要你别把所有东西都硬塞进 Next.js 单进程里。

重点不是“是不是一个仓库”，而是“是不是职责清楚”。

可以是一个 monorepo：

- apps/web：Next.js 主站
- apps/worker：异步任务处理
- apps/runner：Playwright 执行器
- packages/db：数据库 schema 和访问层
- packages/shared：公共类型、常量、工具函数

这样既是一体化管理，又不会把系统写成一坨。

所以“大一统”建议理解成：

一个统一代码仓库和产品形态，
不是一个进程里硬跑所有东西。

## 十三、这套方案最推荐的技术组合

如果让我直接给一个偏务实的组合，我会推荐：

Web 主框架用 Next.js。
数据库用 PostgreSQL。
缓存和队列辅助用 Redis。
异步任务用 BullMQ 之类的队列方案。
测试自动化用 Playwright。
对象存储用公司现有 OSS / S3 兼容存储。
认证走 Microsoft Entra ID。
ORM 可以用 Prisma。
图表层用 ECharts 或 AntV。

这套东西都比较成熟，不会在基础设施上浪费太多时间。

## 十四、这版方案和纯前后端分离相比的优缺点

优点很明显。

开发快，前后端协作成本低，部署链路简单，适合先把产品跑起来。

同时也能保留后续拆分空间，因为 worker 和 runner 本来就是独立的。

缺点也有。

如果未来系统规模很大、团队很多、接口复杂度暴增，Next.js 主应用可能会逐步承担过多责任。

但这是“做大之后再优化”的问题，不是当前阶段最需要担心的事。

对当前阶段来说，先把系统做成、跑通、积累数据，远比一开始就过度拆分更重要。

## 十五、结论

结合你的诉求，这套系统最推荐的落地方式是：

用 Next.js 做一体化主应用，前端页面和大部分服务端逻辑放在一起；
用独立 worker 处理异步分析任务；
用独立 Playwright runner 在无头浏览器里执行测试；
用 PostgreSQL 做中心化持久化；
用 Redis 做队列和缓存；
登录直接接公司微软统一认证。

这套方案既满足“大一统”的开发效率，也满足你要的统计、持久化、权限控制和全局展示能力。

如果继续往下推进，下一步就可以直接细化成数据库表设计和模块拆分清单了。
