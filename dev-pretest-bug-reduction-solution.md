# 面向研发自测阶段的提测 Bug 减少方案设计

创建时间：2026-03-21
项目目录：/home/delta/workspace/ai/testmind

## 一、方案目标

这个项目的最终目标，不是单纯生成 E2E，也不是做一个“AI 写测试”的演示工具，而是帮助开发在提测之前更早发现问题，减少低级 Bug 流入测试阶段。

所以整个方案要围绕一个核心目标来设计：

在开发提测前，基于 Jira、代码变更和项目测试资产，自动生成可执行的自测建议、测试草稿和问题反馈，帮助开发在自测阶段提前暴露问题，从而降低提测 Bug 数量。

一句话概括就是：

把“开发写完代码后靠自己随手点一点”升级成“有结构化引导、有自动验证结果、有问题摘要反馈”的研发自测能力。

## 二、真正要解决的问题

从现有几份文档看，之前的讨论有两个方向：

一个是 Jira 到 E2E 的自动生成，另一个是给老板看前后对比和 Dashboard。

但如果把项目目标收敛到业务价值本身，最核心的问题其实只有一个：

为什么开发提测时还会带进很多本来可以提前发现的 Bug？

根因通常有这些：

开发完成需求后，主要验证的是自己最熟悉的主流程。
很多边界条件、异常分支、权限差异、状态切换和旧功能回归点，没有被系统性检查。

同时，Jira 里的需求信息和代码变更信息并没有自动转成“开发该怎么自测”的具体行动项。
结果就是：

- 开发知道自己应该测，但不知道该系统测什么
- 手工整理测试点费时间，所以常常只做粗略验证
- 真正容易出问题的地方，往往在提测后才被测试同学发现
- 测试发现的问题里，有相当一部分本来应该在开发阶段拦住

所以，这个项目真正要做的，不是替代测试团队，而是给开发补上一层“提测前智能自测防线”。

## 三、方案定位

这个应用的定位建议非常明确：

它是一个“研发提测前质量拦截助手”，而不是一个通用测试生成器。

它的工作重点不是追求全自动生成完美脚本，而是帮助开发在提测前完成三件最有价值的事：

- 知道这次改动最该测什么
- 能快速拿到可执行的自测草稿和验证路径
- 一旦跑出问题，能快速看到失败摘要并回到代码修复

也就是说，它服务的是开发提测前的最后一公里质量控制。

## 四、统一后的核心设计思路

从现有材料里可以收敛出一条更合理的链路：

Jira 需求 + 代码变更 + 项目测试资产 -> 生成结构化自测方案 -> 生成可执行测试草稿 -> 触发自动验证 -> 输出问题摘要 -> 开发修复后再提测

这里最重要的是，输入不能只靠 Jira 文本。

如果只看 Jira，系统很容易“脑补”。
真正有效的输入应该至少包含三类上下文：

第一类是需求上下文，也就是 Jira 的标题、描述、验收标准、父任务、业务关键词。

第二类是变更上下文，也就是 PR diff、改动文件、接口改动、页面改动、组件改动、路由变化。

第三类是测试上下文，也就是已有 E2E、page object、data-testid、测试账号、测试数据准备规则、常见断言方式。

只有把这三类上下文合起来，系统才有机会产出真正对开发有帮助的自测结果。

## 五、系统应该输出什么

如果目标是帮开发减少提测 Bug，那么输出就不应该只是一份测试代码。

更合理的输出应该分成四层。

### 1. 提测前自测清单
这是最基础也是最实用的一层。

系统根据 Jira 和代码改动，先给开发一份结构化自测清单，告诉他这次应该重点验证哪些页面、哪些操作、哪些状态变化、哪些异常情况、哪些回归影响点。

这份清单的价值很大，因为很多时候开发不是完全不测，而是不知道还要补测什么。

### 2. 可执行测试草稿
在自测清单基础上，系统再生成 Playwright 之类的测试草稿。

这里不追求一步到位全自动，而是追求“足够接近可用”，让开发修改成本足够低。

对于常见场景，开发甚至可以直接运行；
对于复杂场景，也至少不用从零开始写。

### 3. 自动执行结果
系统把生成的草稿或者已有测试组合跑起来，给出执行结果。

重点不是只显示 pass/fail，而是要尽量把失败现场保留下来，比如截图、失败步骤、报错日志、关键接口失败信息。

### 4. 面向开发的问题摘要
这是整个方案非常关键的一层。

开发并不想看一大堆原始日志，他最需要的是一句人话：
哪里失败了、可能是什么问题、应该先看哪段代码。

所以系统应当把测试失败结果整理成开发可快速理解的问题摘要，帮助他在提测前完成修复。

## 六、建议的产品流程

一个完整但现实的流程，建议长这样：

开发准备提测时，把 Jira 和当前 PR 或代码分支送进系统。
系统先识别本次改动影响了哪些页面、组件、接口和交互路径。
再结合已有测试资产，生成一份提测前自测建议。

如果命中已有自动化资产，就优先推荐复用已有测试；
如果没有，就生成新的测试草稿。

接着系统触发自动执行，跑完后把结果汇总成开发可读的问题摘要。

如果全部通过，开发更有信心提测；
如果失败，开发就在提测前先修问题。

这一套流程的价值，不在于每次都能生成多高级的测试，而在于它能把“提测前该不该再测一下”从主观习惯，变成一个标准动作。

## 七、方案的关键能力模块

为了支撑这条链路，系统建议拆成六个能力模块。

### 1. Jira 需求解析模块
负责把 Jira 文本转成结构化对象。

这里要抽取的重点不是写得多漂亮，而是能提炼出：

- 这次改动的业务目标是什么
- 用户会做什么操作
- 预期结果是什么
- 哪些地方可能有边界和异常
- 涉及哪个业务域

它的作用是把“描述”转成“可测试语义”。

### 2. 变更影响定位模块
这是整个方案最重要的模块之一。

它要根据 PR diff、改动文件、接口变化、路由和组件索引，判断这次改动最可能影响哪些页面和操作路径。

这个模块越准，后面的测试建议就越靠谱。

如果没有这一层，系统就还是在猜。

### 3. 自测建议生成模块
这层不直接写测试代码，而是先产出一份开发能看懂的自测建议。

建议内容可以包括：

- 本次重点验证页面
- 主流程检查项
- 异常流程检查项
- 权限和状态检查项
- 可能被影响的旧功能
- 建议复用的已有测试

这是最容易真正落地并立刻产生价值的部分。

### 4. 测试草稿生成模块
在有较高置信度时，生成 Playwright 测试草稿。

这里必须模板化，不要让系统自由发挥。

例如统一约束：

- 使用项目已有测试框架
- 使用统一登录流程
- 使用统一测试数据准备方法
- 优先使用稳定 selector
- 统一断言格式
- 标记脚本来源于哪个 Jira 和哪个变更

这样生成结果才有机会稳定接入现有工程体系。

### 5. 执行与归因模块
运行测试后，负责收集结果并做初步归因。

它不需要承诺 100% 判断准确，但至少应该帮助开发区分：

- 是页面真实行为不符合预期
- 是接口返回异常
- 是环境/数据问题
- 是 selector 不稳定
- 是登录态或权限问题

这个模块的目标是减少开发排查时间。

### 6. 质量数据沉淀模块
每次分析、自测建议、脚本生成、执行结果和问题摘要，都应该被结构化保存。

因为长期来看，项目的价值不仅在于一次次运行，而在于不断积累：

- 哪类 Jira 最容易出提测 Bug
- 哪些模块最需要自测补强
- 哪些测试建议最常被采纳
- 哪些失败模式反复出现

这些数据会反过来提升系统准确率，也能支撑后续 dashboard 展示。

## 八、最小可行版本建议

如果要快速落地，我建议 MVP 不要一上来追求很重。

最小闭环应该是：

输入 Jira + PR diff，系统输出提测前自测清单，并为高频场景生成 Playwright 草稿，随后自动执行已有测试或新草稿测试，最后输出问题摘要。

这个 MVP 已经足够验证最关键的业务价值：

能不能在开发提测前多拦住一批明显 Bug。

在这个阶段，先不要追求：

- 全场景自动生成稳定脚本
- 全自动精确归因
- 所有 Jira 都能处理
- 完整老板 dashboard 一次到位

先把“减少提测 Bug”这个主目标打透，比堆很多功能更重要。

## 九、这个方案成功的核心衡量指标

既然最终目标是帮开发减少提测 Bug，那么 KPI 也应该围绕这个目标，而不是围绕“生成了多少脚本”。

最关键的指标建议是：

提测后由测试发现、且本可在开发自测阶段暴露的问题数量，是否明显下降。

这是第一核心指标。

配套可以看几个辅助指标：

- 开发提测前自测覆盖率是否提高
- 系统生成的自测建议采纳率是否提高
- 进入提测前自动验证流程的需求占比是否提高
- Jira 到首轮自测结果的时间是否缩短
- 高风险模块的提测问题密度是否下降

如果这些指标改善，就说明项目方向是对的。

## 十、如果后面要做 Dashboard，应该围绕什么展示

如果后续要做 dashboard，也应该围绕“减少提测 Bug”来设计，而不是围绕模型能力本身。

最值得展示的是这些内容：

第一，提测前拦截了多少问题。
这最能直接证明价值。

第二，哪些模块或需求类型最容易出现提测问题。
这能帮助团队识别薄弱点。

第三，系统给出的自测建议中，有多少被采纳并真正执行。
这能衡量工具是否真的被开发使用。

第四，哪些问题类型最常出现，比如状态更新异常、按钮不可点、权限缺失、接口异常、校验不一致。
这有助于后续做专项治理。

第五，从 Jira 到自测结果产出的平均时长。
这代表方案有没有拖慢研发节奏。

所以 dashboard 是结果层，不是本轮方案的起点。
要先围绕业务目标把链路跑通，再做可视化放大价值。

## 十一、目前还缺哪些关键信息

在现有文件基础上，已经足够收敛出方向，但如果后面要继续细化实施方案，我建议尽快补这几类信息。

### 1. 目标项目的测试栈
需要明确现在实际使用的是不是 Playwright，还是 Cypress，或者项目里压根没有成熟 E2E。

因为这决定测试草稿如何生成，以及能复用多少现有资产。

### 2. Jira 字段和提测流程
需要知道 Jira 里哪些字段稳定可用，比如标题、描述、验收标准、模块标签、关联 PR、提测状态。

同时也要知道开发什么时候提测、谁来触发系统、触发入口在哪。

### 3. 代码变更信息怎么拿
需要明确能否稳定拿到 PR diff、分支 diff、改动文件列表、接口变更信息。

如果拿不到，影响定位能力会大打折扣。

### 4. 项目里有没有可复用测试资产
包括 page object、测试账号、测试数据准备方法、data-testid 规范、已有 E2E 用例。

这部分越完整，系统落地越快。

### 5. 现在“提测 Bug”怎么定义
这个要尽快统一，不然后续 KPI 会飘。

比如到底统计的是：
测试阶段首次发现的问题数，还是被打回给开发的问题数，还是测试阻塞类问题数。

这个定义直接决定项目效果怎么衡量。

## 十二、结论

结合现有全部材料，这个项目最合理的统一定位已经比较清楚了。

它不是一个单纯的 LLM 测试生成工具，也不应该先从老板展示页出发。

它首先应该被设计成一个“研发提测前的质量拦截方案”：

通过 Jira、代码变更和现有测试资产来生成结构化自测建议与测试草稿，并把执行结果转成开发可读的问题摘要，帮助开发在提测前更早发现并修复问题。

只要这条链路跑通，后面的 dashboard、汇报材料、效果展示都会自然成立。

如果这条链路没有跑通，前面的展示做得再漂亮也只是包装。
