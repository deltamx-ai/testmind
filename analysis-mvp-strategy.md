# TestMind 项目分析与 MVP 策略

## 一、项目本质

TestMind 想解决的核心问题：**开发者提测前缺少系统性的自测指引，导致大量本可避免的 bug 流入 QA 阶段。**

解决路径：Jira 需求 + PR diff → LLM 分析 → 生成自测建议 + 测试草稿 → 自动执行 → 结果反馈。

这个方向是对的。但现有文档存在**过度设计**的问题。

## 二、对现有文档的判断

### 做对了的部分
- 问题定义清晰，痛点真实
- LLM 自动采集上下文（而非让开发者手动 prompt）的思路正确
- "自测建议"优先于"自动生成完整测试"的务实定位
- 两级 Dashboard 的分层思路合理

### 过度设计的部分
- **技术栈过重**：MVP 阶段同时引入 Next.js + NestJS + PostgreSQL + Redis + BullMQ，对一个还没写第一行代码的项目来说太多了
- **Dashboard 规划过早**：在没有数据积累之前，设计精细的统计面板没有意义
- **权限体系过早**：四种角色（开发者/项目 Lead/QA Lead/平台 Admin）是成熟期的事
- **测试执行层独立部署**：MVP 不需要分布式 runner 架构

## 三、Electron 要不要用？

**结论：不用。** 但理由和文档里说的不完全一样。

文档说"因为需要跨项目统计所以要 Web"——这是对的但不是最关键的。最关键的原因是：

1. **Electron 会大幅增加工程复杂度**，打包、自动更新、跨平台兼容、本地数据同步，每一项都是坑
2. **核心价值在服务端**（LLM 调用、Jira/Git 集成、测试执行），前端只是展示层，没有离线使用场景
3. **团队协作天然需要 Web**，不存在"个人工具"的使用场景
4. 如果未来需要桌面体验，用 PWA 或浏览器通知即可，不需要 Electron 的重量级方案

唯一可能考虑 Electron 的场景是：需要深度集成本地 IDE。但这件事用 VS Code Extension 做更合适，跟 Electron 无关。

## 四、真正的最小 MVP

文档里定义的 MVP 仍然太大。一个真正可执行的最小 MVP 应该是：

### MVP-0：验证核心假设（1-2 周）

**目标**：验证 "LLM 能否根据 Jira + diff 生成有用的自测建议"

**做法**：
- 一个 CLI 工具或简单脚本
- 输入：Jira ticket ID + PR/branch
- 自动拉取 Jira 描述 + git diff
- 调用 LLM（Claude API）生成结构化自测清单
- 输出到终端 / Markdown 文件

**不做**：Web UI、数据库、队列、Dashboard、认证、Playwright 执行

**为什么**：如果 LLM 生成的自测建议质量不行，后面所有东西都没意义。先验证这个核心假设。

### MVP-1：最小可用产品（4-6 周）

在 MVP-0 验证通过后：

```
┌─────────────────────────────────────────┐
│  Next.js 全栈应用（一个项目搞定）         │
│                                         │
│  页面：                                  │
│  - 登录页（先用简单 token，别急着接 SSO） │
│  - 项目列表 + 配置页                     │
│  - 自测分析页（输入 Jira → 看结果）       │
│  - 执行记录列表                          │
│                                         │
│  后端：                                  │
│  - Next.js API Routes（不需要 NestJS）   │
│  - SQLite（不需要 PostgreSQL）           │
│  - 内存队列（不需要 Redis + BullMQ）      │
│                                         │
│  核心流程：                               │
│  Jira API → diff 获取 → LLM 分析        │
│  → 自测建议 + 可选测试草稿               │
│  → 可选：本地 Playwright 执行            │
└─────────────────────────────────────────┘
```

**关键原则**：
- **一个 Next.js 项目**，不拆微服务
- **SQLite** 够用就行，迁移到 PostgreSQL 很容易（用 Prisma/Drizzle ORM）
- **认证先简化**，等真的要给团队用再接 Entra ID
- **不做 Dashboard**，等积累了 2-3 个月数据再说

### MVP-2：团队可用版（MVP-1 后 4-6 周）

- 接入 Microsoft Entra ID
- 迁移到 PostgreSQL（如果 SQLite 扛不住）
- 加入 BullMQ 异步队列（如果同步执行太慢）
- 基础 Dashboard（项目维度的通过率、拦截数）
- CI 集成（PR 创建时自动触发分析）

## 五、技术选型建议

| 层面 | MVP 阶段 | 成熟阶段 |
|------|----------|----------|
| 框架 | Next.js（App Router） | 同左 |
| UI | Shadcn/ui（轻量、可定制） | 同左或 Ant Design |
| 数据库 | SQLite + Drizzle ORM | PostgreSQL + Drizzle ORM |
| 队列 | 内存队列 / 简单 setTimeout | BullMQ + Redis |
| LLM | Claude API | 同左，可加模型切换 |
| 测试执行 | 本地 Playwright | 独立 Runner 节点 |
| 认证 | 简单 JWT | Microsoft Entra ID |
| 部署 | 单机 Docker | Docker Compose / K8s |

## 六、风险与建议

### 最大风险
1. **LLM 输出质量不稳定**：自测建议可能太泛、不准确。需要大量 prompt 工程和领域知识沉淀
2. **开发者不愿意用**：如果流程太重，开发者会绕过去。必须做到"一键触发、30 秒出结果"
3. **Jira 数据质量差**：很多团队的 Jira 描述写得很烂，LLM 巧妇难为无米之炊

### 关键建议
1. **先做 MVP-0**，花一两周验证 LLM 分析质量，别急着搭架子
2. **从一个项目开始**，找一个配合度高的团队试点，别一上来就想覆盖全公司
3. **自测建议比自动测试重要**——前者帮助开发者思考，后者容易产生不稳定的测试脚本
4. **采集开发者反馈**——每条建议加个"有用/没用"按钮，用来迭代 prompt 和改进上下文策略
5. **不要追求完美的测试生成**——生成的 Playwright 测试能跑通 60% 就很好了，重点是给开发者一个起点

## 七、总结

| 维度 | 文档方案 | 我的建议 |
|------|----------|----------|
| 产品形态 | Web 平台 | ✅ 同意，不用 Electron |
| MVP 范围 | 偏大（含 Dashboard、多角色） | 砍到只做核心流程 |
| 技术栈 | Next.js + NestJS + PG + Redis | Next.js 单体 + SQLite |
| 第一步 | 搭建平台骨架 | **先写个 CLI 验证 LLM 效果** |
| 认证 | 一开始就接 Entra ID | 先简单 JWT，后面再接 |
| 时间线 | 未明确 | MVP-0: 2周, MVP-1: 6周, MVP-2: 6周 |

**一句话**：先用最轻的方式证明 LLM 能生成有价值的自测建议，再考虑平台化。
