# Alma 设计分析与可借鉴架构方法

创建时间：2026-03-22
项目目录：/home/delta/workspace/ai/testmind

## 一、这份分析的目的

这份文档先不急着直接给新系统下结论，而是先把 Alma 这套设计里值得借鉴的地方拆出来。

因为你这次不是单纯问“要不要用 Electron”或者“表怎么设计”，而是希望参考电脑上的 Alma，先分析它为什么这样设计，再反过来指导你现在这个研发自测工具怎么做。

这个顺序是对的。

真正能复用的，通常不是某个具体表名或者某个技术栈，而是背后的设计方法。

## 二、从当前机器能观察到的 Alma 设计特征

从本机可见信息来看，Alma 不是一个只会聊天的单点程序，而是一套“桌面承载 + 本地持久化 + CLI 管理 + 插件扩展 + 任务编排 + 多能力入口”的综合系统。

也就是说，Alma 的设计重点并不只是 UI，而是把很多能力收进一个统一的本地产品框架里。

从可见结构上，大概能总结出几类特征。

### 1. 本地优先的数据与配置组织
Alma 把很多长期状态和配置都放在 ~/.config/alma 下面。

这意味着它天然把“本地持久化”和“用户私有状态”当作一等公民，而不是完全依赖远端服务。

像这些目录就很典型：

- SOUL.md
- USER.md
- MEMORY.md
- HEARTBEAT.md
- memory/
- people/
- groups/
- chats/
- reports/
- skills/
- plugins/
- missions/
- tasks/

这说明 Alma 的状态不是只存在内存里，也不是只在一个数据库里黑箱保存，而是采用“本地文件 + 结构化状态”混合设计。

这个思路很适合你现在要做的隐私优先工具。

### 2. 能力入口很多，但底层状态是统一的
Alma 同时暴露了：

- 桌面应用
- CLI
- 配置系统
- 浏览器能力
- 线程系统
- memory 系统
- skill 系统
- mission 系统
- cron/heartbeat 系统

入口很多，但从表现上看，它们不是各玩各的。

它们共享同一套配置、状态、任务和数据目录。

这点很重要。

因为它说明一个成熟本地产品，不一定只有一个入口；
真正重要的是多个入口背后是否共享统一核心。

### 3. 任务化、编排化很明显
从 missions.json 和 tasks.json 可以看出来，Alma 不是“收到请求就立刻做完然后消失”的简单交互程序。

它有明显的任务拆解、状态推进、事件记录和阶段性完成机制。

这背后代表的是一个很重要的设计思想：

把复杂工作从“即时动作”提升成“可追踪任务”。

这对你想做的研发提测前检查工具非常有借鉴意义。

因为“提测前检查”本身就不是单一步骤，而是一条链路：

读需求、取变更、分析影响、生成建议、执行测试、汇总问题。

如果不用任务模型来组织，后面一定会乱。

### 4. 插件和技能是扩展主机制
Alma 明显把 skill 和 plugin 当成扩展能力的主入口，而不是把所有功能都塞在核心里。

这说明它采用的是“核心保持稳定、变化能力外插”的思路。

这也是为什么它可以不断增加：

- 新 provider
- 新 skill
- 新插件
- 新浏览器能力
- 新集成方式

而不至于每次都把主程序改得面目全非。

这对你的系统尤其重要，因为你后面明显也会加：

- 不同需求源
- 不同模型
- 不同执行器
- 不同测试框架
- 不同同步策略

所以借 Alma 的正确方式，不是抄它的 skill 名字，而是借它这种扩展哲学。

### 5. 结构化数据和人类可读文件并存
Alma 既有 missions.json、tasks.json 这种结构化状态，也有 SOUL.md、USER.md、MEMORY.md、chat/group log 这种面向人的文件。

这其实是一个很值得借鉴的平衡点。

纯结构化适合程序处理，但不适合人工观察和手工修正。
纯文本适合人读，但不利于稳定自动化。

Alma 是把两者混合使用。

这意味着它区分了两类数据：

一类是程序必须稳定消费的运行状态；
一类是更适合以文本方式沉淀、审阅和演化的知识与上下文。

你的系统其实也需要这种区分。

比如：

项目、检查任务、执行结果，这些适合结构化存。
而分析摘要、架构报告、人工备注、补充规则，这些完全可以文本化沉淀。

### 6. 本地文件系统是产品能力的一部分，不只是存储位置
Alma 的很多能力天然围绕本地文件和目录工作。

这说明在它的设计里，文件系统不是附属物，而是核心能力载体之一。

这和很多纯 Web 产品很不一样。

对你的场景也一样。

如果是隐私优先 Electron，本地仓库、截图、trace、执行报告、模型上下文缓存，本来就应该是系统核心组成，而不是后面补丁式地挂上去。

## 三、从 Alma 里最值得借鉴的六个设计方法

如果把上面的观察再往抽象层提一层，我觉得最值得借鉴的不是具体实现，而是这六个方法。

### 1. 统一核心，分散入口
Alma 的入口很多，但底层数据和能力核心是统一的。

对你的系统来说也是一样。

以后你可能有：

- Electron 桌面入口
- CLI 入口
- IDE 协作入口
- 可选后台同步入口

这些入口可以同时存在，但都不应该各自维护一套逻辑。

正确做法是：

入口分散，核心统一。

### 2. 任务中心化，而不是动作中心化
Alma 不是单纯执行一串零散命令，而是把工作放进 mission/task 体系里。

这对复杂流程特别有价值。

你的系统完全可以借这个思想，把“提测前检查”设计成一个检查任务对象。

这样每次检查都有：

- 输入来源
- 当前状态
- 已完成阶段
- 产出建议
- 测试执行结果
- 失败摘要
- 最终结论

这样后面做历史追踪、调试、统计都会容易很多。

### 3. 核心数据结构化，边缘知识文本化
Alma 用 json 管理可执行状态，用 md/log 管理更适合人工理解的内容。

这是很实用的组合。

你也应该这么做。

比如：

- Project、CheckTask、TestRun、IssueSummary 放数据库
- 分析说明、设计记录、项目规则、人工评审意见 放 markdown 或文本附件

不要强行把所有东西都塞成数据库表，也不要把所有东西都做成文件。

### 4. 扩展机制前置，而不是后补
Alma 明显不是先写死一坨功能，再想办法补插件。

它从设计上就给 skill、plugin、provider 留了位置。

你的系统如果现在不把扩展点先设计出来，后面一定会出现：

- Jira 写死
- Playwright 写死
- 某个模型 SDK 写死
- 同步策略写死

然后每次扩一个能力都要动核心。

这就是最该避免的事。

### 5. 文件系统和配置系统是正式能力
Alma 不是把配置随便丢几个 json 就完了。

它明显把配置、workspace、thread、memory、reports 都变成系统内部正式对象。

这说明一个本地型产品要做强，必须认真设计：

- 配置层
- 目录层
- 工作空间层
- 结果输出层

你的工具也一样。

比如项目路径、模型策略、隐私模式、测试框架、忽略规则、附件目录、同步策略，都不能靠零碎配置散着放。

### 6. 编排和扩展比单次智能更重要
Alma 的价值并不只是“会回一句话”，而是把很多能力编排起来：

线程、memory、skill、browser、mission、cron、config。

这说明一个复杂 AI 本地产品，真正决定上限的不是某一次模型回答，而是：

- 有没有稳定编排
- 有没有可追踪状态
- 有没有扩展能力
- 有没有长期持久化

你的研发自测工具也是一样。

真正有价值的，不是某次 LLM 一口气写出一个测试，而是整条系统能不能稳定工作。

## 四、哪些地方不要硬照 Alma 抄

讲完能借的，也要讲不能硬抄的。

### 1. 不要照搬对象命名
Alma 有 thread、memory、mission、skill、provider 这些对象，是因为它服务的是一个通用智能桌面产品。

你的业务对象完全不同。

你更核心的对象应该是：

- Project
- RequirementTask
- ChangeSet
- CheckTask
- Recommendation
- TestCase
- TestRun
- IssueSummary

所以可以借抽象方法，不要硬搬名词。

### 2. 不要把一切都做成“聊天式”
Alma 天然有对话线程中心。

但你的系统主线是研发提测前质量检查，不是聊天本身。

所以不要因为参考了 Alma，就把整个产品设计成一个聊天窗口包打天下。

聊天可以是入口之一，但不应该吞掉主业务流程。

### 3. 不要过早复制它的全功能体系
Alma 现在看起来能力很多，是长期演进出来的。

如果你一上来就复制它那种 breadth，会把自己拖死。

正确做法是借它的骨架，不是复制它的体量。

## 五、从 Alma 分析推回你这个系统，最该借的架构骨架

如果把 Alma 的设计方法翻译成你这个项目的语言，我觉得最值得借的骨架是下面这套。

第一，桌面产品只是壳，核心在统一引擎。

第二，所有复杂流程都进任务系统，而不是零散按钮逻辑。

第三，核心实体结构化，附件与文本说明分层存。

第四，数据源、模型、执行器、同步策略都插件化。

第五，本地文件系统和本地数据库都是正式一等公民。

第六，后续入口可以增加，但必须共用一个核心。

## 六、对你当前项目的直接启发

结合你现在“隐私优先、本地优先、Electron、后续可扩展、还能参考 Alma”的目标，我觉得最关键的启发有四个。

### 1. 一开始就做任务对象
不要把“开始分析”“开始执行”“查看问题”这些都做成分散逻辑。

应该先有一个 CheckTask 作为主线对象。

### 2. 一开始就做插件点
哪怕第一版只支持 Jira、Playwright、一个模型，也要先定义接口。

### 3. 一开始就把本地文件目录规划好
数据库不是全部。
截图、trace、日志、分析产物、补充规则文件都要有稳定目录结构。

### 4. 一开始就区分核心状态和派生结果
比如：

核心状态是检查任务、执行记录、建议项、问题摘要。
派生结果是 dashboard 快照、周报、导出文件、共享摘要。

这两类不要混在一起设计。

## 七、结论

如果只看这台机器上能观察到的 Alma 设计，我觉得它最值得借鉴的不是某个具体技术实现，而是下面这套产品工程方法：

本地优先持久化、统一核心、多入口共享、任务中心化、插件扩展、结构化与文本并存、文件系统作为正式能力、复杂流程靠编排而不是靠单次智能回答。

这套方法非常适合迁移到你现在要做的隐私优先研发自测工具里。

所以接下来给你的详细架构设计，不应该只是“做一个 Electron UI + 调一下 Playwright”，而应该按这套方法把系统真正搭起来。
