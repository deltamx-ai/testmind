# Copilot 接入与 LLM 上下文获取策略设计

创建时间：2026-03-21
项目目录：/home/delta/workspace/ai/testmind

## 一、这次问题的核心

你这次问的其实是两个关键点。

第一，前面的方案里为什么没把 Copilot 单独纳进来。

第二，如果把 Copilot 加进来，它到底扮演什么角色；系统是自己自动收集信息交给模型，还是要靠人手动整理后再喂给模型。

这两个问题其实决定了整个方案到底是“AI 编辑器辅助”，还是“平台级自动分析系统”。

## 二、先说结论

Copilot 可以加入，而且值得加入，但不应该把它当成唯一核心能力。

更准确地说，Copilot 更适合放在“开发本地交互层”和“代码草稿生成层”，而平台本身仍然需要负责自动收集上下文、整理结构化信息、调度模型和沉淀结果。

也就是说：

Copilot 可以帮助开发更快生成和修改测试；
但不能把整套信息收集、任务编排、结果持久化、项目统计都寄托在 Copilot 身上。

所以正确的定位是：

平台负责自动拿信息和组织上下文；
Copilot 负责帮助开发在 IDE 里更高效地消费这些信息并产出最终代码。

## 三、为什么前面没把 Copilot 放成核心

因为你前面讨论的重点已经不只是“怎么让模型写出测试”，而是：

- 怎么减少提测 Bug
- 怎么做跨项目统计
- 怎么沉淀测试执行结果
- 怎么做 dashboard
- 怎么接入统一认证
- 怎么让平台持续积累质量数据

这些问题本质上都超出了 Copilot 的边界。

Copilot 本质更像一个强大的开发辅助入口，它很适合帮助开发：

- 在 IDE 里生成一段测试代码
- 根据当前文件上下文补全 Playwright 脚本
- 修改失败用例
- 根据报错快速修断言或 selector

但它不天然负责：

- 跨系统拉 Jira
- 拉 PR diff
- 建项目级上下文
- 做统一统计面板
- 存执行结果
- 做组织级权限和可视化

所以前面的方案不是“忘了 Copilot”，而是前面的重点是平台能力，Copilot 只是其中一个可接入入口。

## 四、Copilot 在这套方案里最合理的角色

如果把整个系统拆开看，Copilot 最适合承担三类角色。

### 1. 本地测试草稿生成助手
开发在 IDE 中查看某个 Jira、某次变更或者平台推送下来的自测建议时，可以直接让 Copilot 帮他生成 Playwright 测试草稿。

这个场景非常自然，因为 Copilot 本来就离代码最近。

比如平台已经判断出：

- 受影响页面是订单详情页
- 重点检查点是状态流转和权限按钮显示
- 推荐复用现有 order-detail.spec.ts

那开发在 IDE 里就可以基于这些结构化信息，让 Copilot 快速补出测试代码。

### 2. 测试修复助手
当平台或 runner 跑出失败结果后，开发可以把失败摘要、日志、截图线索和当前测试文件一起交给 Copilot，让它帮助：

- 修 selector
- 补等待条件
- 调整断言
- 修复测试数据准备逻辑

这一步 Copilot 的效率会很高，因为它本来就擅长围绕当前代码上下文做修改。

### 3. 开发自测问答入口
如果开发想在本地快速问：

“这次 PR 最应该补哪些测试？”
“这个失败更像是真 Bug 还是脚本问题？”
“基于现有 page object 帮我补一个 case。”

那 Copilot 也很适合做这种交互层。

但它要想回答得准，前提还是平台先把上下文准备好。

## 五、Copilot 能不能直接拿来做测试提取和生成

可以，但要分情况。

### 适合 Copilot 直接做的部分
这些事情 Copilot 很适合：

- 根据当前测试文件补全 case
- 根据 page object 生成一段测试流程
- 根据已有 selector 补断言
- 根据失败日志帮你修改脚本
- 根据局部 diff 推测需要补的测试点

因为这些任务都强依赖本地代码上下文，而且交互周期短。

### 不适合只靠 Copilot 的部分
这些事情如果只靠 Copilot，就会比较弱：

- 自动拉取 Jira 并结构化分析
- 在多个项目之间统一收集统计信息
- 自动保存每次分析和执行结果
- 统一生成 dashboard 数据
- 给管理层看整体趋势
- 做跨人、跨项目、跨时间的质量沉淀

这些必须由平台来做。

所以答案不是“要不要 Copilot”，而是“Copilot 放在哪一层最合适”。

## 六、自动获取信息，还是手动整理给 LLM

结论也比较明确。

应该以“系统自动获取信息”为主，手动补充为辅。

如果核心信息都要开发手动整理后再喂给模型，这个系统会很难真正被持续使用。

因为开发最不愿意做的事情，就是多一道人工整理动作。

你想解决的是提测前质量问题，那就要尽量减少开发额外负担。

所以理想路径应该是：

系统自动去拿 Jira、PR diff、改动文件、已有测试资产、历史执行结果；
开发只在系统信息不足时补充少量上下文，比如：

- 这次改动的特殊前置条件
- 需要使用哪个测试账号
- 某个新页面还没录入路由信息
- 某个需求有额外验收约束

也就是说，默认自动收集，必要时人工补洞。

## 七、为什么不能主要依赖手动喂模型

因为手动模式有几个明显问题。

第一，不稳定。
不同开发补充的信息质量差异很大，系统输出结果也会飘。

第二，不可规模化。
项目少的时候还能忍，项目多了、需求多了，就没人愿意持续手动整理。

第三，不可统计。
如果上下文来源太随意，后面很多分析结果都难以标准化沉淀。

第四，体验差。
你本来是想减少开发负担，结果变成开发每次提测前还要先当一遍“提示词工程师”，这就跑偏了。

所以从产品设计上，手动输入应该是兜底，不应该是主路径。

## 八、最合理的上下文策略

我建议上下文获取分三层。

### 第一层：系统自动采集的刚性上下文
这是必须自动拿的。

包括：

- Jira 标题、描述、验收标准
- PR diff、改动文件、commit 信息
- 路由和页面索引
- 现有测试文件
- page object
- data-testid
- 最近执行结果
- 项目测试框架配置

这部分不应该让人手动搬运。

### 第二层：系统自动推理出的中间上下文
这是平台分析后生成的结构化信息。

包括：

- 候选受影响页面
- 候选测试点
- 候选回归风险点
- 推荐复用的已有 case
- 高风险模块判断

这部分是平台的核心价值之一。

### 第三层：人工补充上下文
只有在自动信息不足时，才让人补。

比如：

- 新功能还没上线完整路由
- 页面需要特定灰度开关
- 测试账号需要特殊权限
- 业务上存在代码里看不出来的隐藏规则

这样人工输入就变成增强项，而不是主工作流。

## 九、Copilot 和平台怎么配合最顺

最顺的方式不是二选一，而是两者分工。

平台负责：

自动采集 Jira、代码变更和测试资产；
自动形成结构化自测建议；
自动沉淀执行结果和统计数据；
自动给出失败摘要和项目级质量视图。

Copilot 负责：

在开发本地根据平台提供的上下文，快速生成或修改测试代码；
围绕现有代码做交互式补全；
帮助开发更快消费平台输出。

如果以后要做得更自然，甚至可以让平台提供一个“复制到 Copilot 的上下文包”或者 IDE 插件桥接能力。

也就是说，平台做“大脑和记忆”，Copilot 做“手边的写代码助手”。

## 十、是否可以支持多个模型，包括 Copilot

可以，而且建议系统从架构上就按“多模型接入”设计。

因为未来很可能不是只有一个模型来源。

你可以把模型层抽象成统一接口，然后支持：

- 平台内置 LLM
- Azure OpenAI / OpenAI
- 公司内部模型
- GitHub Copilot 对应的交互能力
- 其他代码生成模型

但要注意一点。

Copilot 和通用 LLM 不完全一样。

Copilot 更偏 IDE 内联协作；
平台模型更偏服务端批处理和结构化分析。

所以它们不是互相替代，更像互补。

## 十一、这一版方案最推荐的做法

如果让我直接给建议，我会这么定。

平台层继续按 Next.js 一体化方案走，负责：

- 自动拿 Jira、PR、测试资产
- 统一持久化数据
- 统一跑分析和测试任务
- 统一展示 dashboard

模型层按可插拔设计，支持多个模型来源。

开发使用层允许接入 Copilot，主要用于：

- 测试草稿生成
- 本地脚本修复
- 自测问题问答

上下文策略上，明确坚持：

自动采集为主，人工补充为辅。

这样既能保证平台化能力，又不会放弃 Copilot 带来的本地生产力。

## 十二、结论

Copilot 不是没法加入，而是不能把它误当成整套系统本身。

它非常适合加入，而且会明显提升开发在本地生成和修改测试的效率。

但整套“减少提测 Bug”的平台能力，仍然必须由系统自动去收集信息、组织上下文、调用模型、保存结果、沉淀统计。

所以最终最合理的方案是：

平台自动拿信息并做统一分析，Copilot 作为开发侧增强入口参与测试生成和修复；
默认自动给模型提供上下文，不以人工手动喂信息作为主路径。
